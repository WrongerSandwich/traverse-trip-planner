# Reconcile brochure and itinerary

**Date:** 2026-05-16
**Status:** Approved
**Origin ticket:** [#133](https://github.com/WrongerSandwich/traverse/issues/133)
**Related:** [#112 lifecycle simplification design](2026-05-16-lifecycle-simplification-design.md) — this design picks up where that arc left off.

## Problem

After the lifecycle-simplification arc (#112 / #118–#123), Traverse has two parallel AI-generated artifacts per planning trip:

- **`brochure.md`** (YAML) — generated by Prepare brochure. Five top-level sections:
  - `days[]` — structured day-by-day: blocks per period (morning / afternoon / evening), items with time + activity
  - `stops[]` — places with category, address, hours, notes, geocoded coords
  - `lodging[]` — overnight records with night counts and coords
  - `field_guide_notes[]` — trip-level reference
  - `gotchas[]` — caveats
- **`itinerary.md`** (Markdown) — generated by the streaming "Generate itinerary" flow. One thing: the day-by-day, in prose form (`## Day N` / `### Morning` / `- 9:00 AM — …`).

The crux: **`brochure.days` and `itinerary.md` encode the same content in different formats.** Brochure's structured `days[]` maps directly to the same prose the streaming flow produces. The other four brochure sections (stops, lodging, field guide, gotchas) are unique. So `itinerary.md` is a strict subset of `brochure.md`.

Both have:
- Their own staleness signal (post-#129 / #132).
- Their own print path (brochure has its print-optimized route at `/trips/<slug>/brochure`; itinerary prints from the detail page via #128's button).
- Their own AI call.
- Their own feature slot (`itinerary` in `FEATURE_SLOT`; brochure-prepare is implicit in the brochure prepare endpoint).

This was a side effect of the pre-#120 lifecycle where "Lock trip" meant freezing the trip and synthesizing an itinerary. With locking demoted (#120), the streaming-itinerary action no longer has a state-machine role — it just produces a redundant artifact.

## Goals

1. One source of truth for the day-by-day content.
2. One AI generation step that produces the day-by-day plus the structured reference data.
3. One printable artifact.
4. Preserve content for any trip that already has an `itinerary.md` on disk.

## Non-goals

- Changing the brochure's content model (still YAML with the same five sections).
- Changing the brochure prepare workflow (Ambient Background, Confirm modal, structured review form).
- Re-deriving the lifecycle.

## Approach

**Brochure is the canonical artifact. Retire `itinerary.md` as a separate generation target.**

### Reading and rendering

`brochure.days` becomes the source for the day-by-day view that today lives in the itinerary tab.

- **Trip detail page** (`/trips/<slug>`): the slot currently rendering `itinerary.md` instead renders the day-by-day view derived from `brochure.days` when `brochure.md` exists. Same visual placement, same Print button (#128). Print routes to the brochure's print layout (`/trips/<slug>/brochure`) rather than the detail page — the brochure layout was always the better print target.
- **Brochure component** (`Brochure.svelte` at `/trips/<slug>/brochure` and `/share/<token>/brochure`): unchanged. It already renders the structured brochure when present.
- **Share view**: unchanged. Already renders itinerary if present; the migration step below preserves backward compatibility.

### Generation

- "Generate itinerary" button on the trip detail page is removed.
- `POST /api/itinerary/[slug]` endpoint is removed.
- The In-Page Stream UX for itinerary on the detail page is removed (streaming banner, preview pre, state machine).
- "Prepare brochure" (Ambient Background) becomes the sole generator for day-by-day content. No UX change to brochure prepare itself.

### Editing

- Direct prose editing of an itinerary file goes away.
- Edits to day-by-day content happen via the brochure prepare/review form (`/trips/<slug>/brochure/prepare`) — same structured editing surface that already exists.
- Section-level edits (overview, route, stops, logistics) still happen on the detail page and trigger brochure staleness as usual.

### Migration for trips with existing `itinerary.md`

Some trips will have an `itinerary.md` but no `brochure.md` (the user generated an itinerary at some point but never prepared a brochure). Do not auto-convert.

Behavior for these trips:
- Detail page renders the existing `itinerary.md` read-only in the same slot it occupies today.
- A "Prepare brochure to enable editing" CTA appears below the read-only itinerary, linking to the prepare flow.
- After brochure is prepared, the `itinerary.md` becomes ignored (the detail page renders `brochure.days` from then on). The file stays on disk — don't delete user data — but no UI surface displays it.

For trips with **both** `itinerary.md` and `brochure.md`: render the brochure view; ignore the itinerary file. The brochure is newer if it was prepared after the lifecycle change.

For trips with neither: the detail page shows no day-by-day view (current behavior).

### Staleness

- `isItineraryStale` and the corresponding UI affordance are removed.
- `isBrochureStale` (post-#132) stays — it's the only artifact-staleness signal now.
- `isArtifactStale` helper stays as the parameterized core.

### Feature slot and env

- `HAND_DEFAULTS.itinerary` is removed.
- `FEATURE_SLOT.itinerary` is removed.
- The `TRAVERSE_MODEL_ITINERARY*` env vars no longer have a slot.
- The `TRAVERSE_MODEL_LOCK*` deprecation warning from #131 becomes obsolete — both the warning and the slot are gone in the same PR.

### Telemetry

- `data.promises.itinerary` is removed from the layout server load.
- The `'itinerary'` key in `.workflow-stats.json` becomes orphaned. Future workflow-stats GC (out of scope here; could be its own ticket if desired) would drop it; until then it's harmless.

## Affected surface

- **Endpoint:** `src/routes/api/itinerary/[slug]/+server.js` (delete).
- **Detail page** (`src/routes/trips/[slug]/+page.svelte`): remove the itinerary state machine (`itinState`, `itinStreamingText`, `itinTokens`, `itinErrorCode`, `itinElapsed`, `itinAborter`, `itinElapsedTimer`, `generating` derived, all `itin*` functions and the StreamBanner block). Remove the Generate itinerary button. Remove the itinerary staleness notice. Add the brochure.days renderer in the existing itinerary slot.
- **Loader** (`src/routes/trips/[slug]/+page.server.js`): drop `itineraryStale`; load brochure data so `brochure.days` is available for rendering. The brochure data lives at `<dir>/brochure.md` and parses as YAML — there's already a brochure-reading helper in `src/lib/server/brochure.js` (see `readBrochure`).
- **Data layer** (`src/lib/server/data.js`): drop `isItineraryStale` wrapper. Keep `isArtifactStale` and `isBrochureStale`.
- **Config** (`src/lib/server/config.js`): drop `itinerary` from `FEATURE_SLOT`. Drop the `warnOnDeprecatedEnv` invocation entirely (the warning was a stopgap for the rename; with both names retired, the warning is dead).
- **Promises** (`src/lib/server/promises.js`): drop `HAND_DEFAULTS.itinerary`.
- **Brochure reuse** (`src/lib/server/brochure.js`): currently does `...getEffectiveConfig().features.itinerary,` to borrow the model slot. Switch to `features.brochure` (a new slot for the brochure-prepare model) or `modelDefault` (simpler — let the brochure inherit the default unless overridden). Pick `modelDefault` for now; a per-brochure slot can land later if anyone needs it.
- **Tests:** delete `tests/api-itinerary.test.js`. Update `tests/config.test.js` (drop itinerary slot + warning tests). Update `tests/data-artifact-stale.test.js` (drop the `isItineraryStale` wrapper test). Update `tests/workflow-stats.test.js` if it still references `'itinerary'` as a label key.
- **Share view** (`src/routes/share/[token]/+page.svelte`): currently prefers `files.itinerary` when present; update to prefer the brochure layout (already wired via `Brochure.svelte`). The share view's brochure layout already exists at `/share/<token>/brochure` — consider whether the bare `/share/<token>` page should redirect/render the brochure layout instead. For this design pass, keep the existing fallback chain: brochure rendered when `brochure.md` exists; `itinerary.md` rendered as legacy fallback; planning sections otherwise.
- **Docs:** `CLAUDE.md` — collapse the "Generate itinerary" and "Print / Save PDF (itinerary tab)" rows into the brochure section. Drop the itinerary-as-artifact framing introduced in #120. Drop the `_promise` itinerary references in `docs/ai-workflow-ux.md` if any remain.

## Implementation order

1. **Render `brochure.days` on the detail page** — additive change. Detail page learns to read brochure data and render its day-by-day view in the existing itinerary slot. Existing itinerary file continues to render when brochure is absent. No removals yet.
2. **Retire the "Generate itinerary" action** — drop the button, the endpoint, the In-Page Stream state machine, the staleness affordance, the `_promise` export, the `itineraryStale` flag. The legacy `itinerary.md` fallback rendering from step 1 stays.
3. **Drop dead code** — `isItineraryStale` wrapper, `FEATURE_SLOT.itinerary`, `HAND_DEFAULTS.itinerary`, `warnOnDeprecatedEnv` (and its tests), `tests/api-itinerary.test.js`, the `data.promises.itinerary` plumbing. Brochure's model slot moves to `modelDefault`.
4. **Docs sweep** — `CLAUDE.md`, `docs/ai-workflow-ux.md`, `PRODUCT.md` if needed.

#1 is the prerequisite for everything. #2 follows. #3 and #4 trail.

## Migration

No on-disk migration. Existing `itinerary.md` files are preserved and rendered read-only as a fallback until a brochure is prepared. After brochure prepare, the file is ignored (still on disk; no UI surface).

## Risks

- **Brochure prep takes longer than itinerary generation today** (~45s ambient vs 20–40s streaming inline). The user gives up the watch-it-stream UX. Mitigation: the brochure flow is already established and the Ambient Background pattern is well-trodden in this codebase.
- **Brochure review form is more constrained than markdown editing.** A user who wants to hand-write prose in their itinerary loses that surface. Mitigation: the brochure's structured editing covers all the day-by-day data; prose can still be edited via the section files (overview/route/stops/logistics).
- **Old `itinerary.md` files become dead weight on disk once a brochure is prepared.** Mitigation: don't delete; just stop rendering. Surface a "Delete legacy itinerary.md" follow-up if anyone cares.
- **The brochure's print layout becomes the only print path.** If anyone wanted to print the bare prose itinerary, they can't anymore. Mitigation: the brochure print layout is strictly better — adds maps, photos, stops — and the day-by-day prose is still in there.
